Wildcard certificate

EZproxy 2.2a (2003-08-02) contained the following change:

When generating a new certificate under proxy by hostname, you now have the option to create a wildcard certificate. This tells EZproxy that it should add "login." to the front of its name when handling login requests and should generate its SSL certificate using the form "*.ezproxy.yourlib.org". It also changes periods to hyphens when generating the host name for sites using https. These changes allow a single wildcard certificate to generate only one error if self-signed, and no errors if purchased from a certificate authority.

Example: With SSL enabled in EZproxy, a URL such as https://www.somedb.com is proxied as https://www-somedb-com.ezproxy.yourlib.org (note there are hyphens and not periods in www-somedb-com). This merging of all domain names below ezproxy.yourlib.org into a single level means the hostname matches *.ezproxy.yourlib.org when the browser takes the asterisk to mean a single level.

Free certificates for educational domains

As of October 2008, ipsCA at http://certs.ipsca.com/ provides certificate authority issued certificates, including wildcard certificates, for no cost to educational domains including .edu, .edu.au and .ac.nz. It is not clear from the ipsCA website that the offer of free certificates extends to wildcard certificates, but it does. It seems the way to get a free wildcard certificate is to apply for an ordinary edu certificate, but make the certificate signing request (CSR) specify a wildcard.

See also the OCLC page on SSL certificate options.

Replacing a private key

It is possible to import and replace the SSL private key used by EZproxy.

Create a self-signed certificate by going to http://yourproxy:2048/ssl. This will create a directory called c:\ezproxy\ssl. Inside, there will be a set of files named:

  • 00000001.key (RSA private key)
  • 00000001.crt (RSA cert)

If you already have an SSL certificate for your ezproxy hostname, you may replace these files (revisiting http://yourproxy:2048/ssl to add CA certificates to the chain, if necessary) and restart EZproxy.

Generating a certificate signing request using SHA-256

With Google announcement that they will gradually phase out SHA-1 hash algorithm for certificate signing [1], comes the need to issue CSR using SHA-256. Versions of EZproxy 5.7.44 and earlier can only generate certificates using SHA-1. To generate a SHA-256 certificate, you must do so outside of EZproxy and import the key and certificate after having it signed by a CA.

Here is a procedure to generate a SHA-256 signed certificate at the Linux command line, using openssl:

  • From a Linux command line:

openssl req -nodes -sha256 -newkey rsa:2048 -keyout filename.key -out filename.csr

  • You'll be prompted for your information for the signing request. For a wildcard cert, prepend *. to your proxy hostname at the corresponding prompt.
  • To verify correct information in the signing request run:

openssl req -text -noout -in filename.csr

  • Send the contents of filename.csr to your CA for signing.
  • To self-sign the certificate do the following (this will generate a certificate that expires in 1 year):

openssl x509 -req -days 365 -in filename.csr -signkey filename.key -out filename.crt

  • You can use any text viewer to view the key, signing request, or certificate. To do so at the command line, you can use cat or less:

cat filename.csr
less filename.csr

SSL Configuration Options

As of EZproxy version 5.7.44, only SSLv2, SSLv3 and TLSv1.0 are available for securing EZProxy, but SSLv3 is disabled by default. It is generally recommended that you additionally disable SSLv2. Both SSLv2 and SSLv3 are insecure in light of recent security advisories. Depending on your user base, you may need to still support one or both of those older protocols, however, most operating systems and browsers released within the past 5 years will work with the following "most secure" configuration options.

Please see the EZproxy Security FAQ link below.

Best practices might include adding the following options BEFORE your LoginPortSSL option. Please refer to the EZproxy reference manual for their meaning:
Option ForceHTTPSLogin
Option ForceHTTPSAdmin
Option DisableSSLv2
#Redundant, as SSLv3 is disabled by default
#Only available as of version 5.7.44
Option DisableSSLv3
LoginPortSSL 443

Qualys SSL Server Test

Qualys hosts a free public SSL testing tool to evaluate server security.

Due to EZproxy's lack of TLS 1.1 and 1.2 support, the highest grade you can get is B.

SSLCipherSuite

This option should also come before the LoginPortSSL option. It controls which encryption ciphers EZproxy will attempt to use. Explanation of specific options can be found in OpenSSL's documentation. The option string uses the same format that Apache httpd and other web browsers use and various configurations can be found around the net.

One such example string can be found here and the option in EZproxy would look like:
SSLCipherSuite ECDH+AESGCM:DH+AESGCM:ECDH+AES256:DH+AES256:ECDH+AES128:DH+AES:ECDH+3DES:DH+3DES:RSA+AESGCM:RSA+AES:RSA+3DES:!aNULL:!MD5:!DSS

I would recommend using the Qualys SSL Server Test after changing SSLCipherSuite and paying attention to the "Handshake Simulation" section to ensure a wide array of browsers are supported.
